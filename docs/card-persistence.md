# Card Persistence and Token Ownership

This document describes the implemented backend foundation for physical EmerCard persistence. It does not add card HTTP endpoints, anonymous lookup, frontend controls, token migration, QR/NFC handling, inventory, scans, or audit UI.

## Domain boundary

A physical card owns its public-access identity. The medical profile remains user-owned and is not duplicated into card documents:

```text
user
  -> one medical profile

user
  -> zero or more cards
       -> stable serial
       -> unique token hash
       -> independent lifecycle
```

A user may have multiple current cards and multiple active cards at the same time. Each card resolves to the same future public profile projection, but each card has a distinct token and lifecycle.

## Identifier contracts

| Identifier | Persistence | Purpose |
|---|---|---|
| `card_id` | BSON `_id` | Internal persistence identity |
| `serial` | Uppercase canonical string | Human-readable operational identity |
| Raw public token | Never persisted | Public QR/NFC secret, returned once by trusted provisioning |
| `token_hash` | `v1$` + 64 lowercase SHA-256 hex characters | Deterministic indexed lookup key |
| `owner_id` | BSON user ID or `null` | Card ownership |

### Serial

Serials use `EMC-XXXX-XXXX-XXXX-C`:

- `EMC` is the fixed prefix.
- The twelve payload characters use uppercase Crockford Base32 (`0-9A-HJKMNP-TV-Z`).
- The checksum is the sum of the payload's Crockford values modulo 32, encoded with the same alphabet.
- Input comparison is case-insensitive; persistence is uppercase canonical form.
- Serials are generated by the system and must not encode user IDs, timestamps, medical data, or token material.
- The unique MongoDB index is the final collision authority.

The checksum algorithm is intentionally local to `src/emercard/modules/cards/identity.py`; it must be confirmed before externally manufactured inventory is printed.

### Public token

Each provisioned card receives exactly 32 cryptographically secure random bytes encoded as unpadded Base64URL. The exact raw token string is hashed as UTF-8 with SHA-256 and stored as:

```text
v1$<64-character-lowercase-sha256-hex>
```

The raw token is returned only after successful persistence through the trusted `CardService.provision_unassigned` or replacement flow. It is never stored, logged, included in exceptions, or returned from ordinary card reads. Activation, disablement, and assignment never rotate it. Replacement creates a new token for the new physical card.

## Card document

`CardDocument` contains:

```text
_id
serial
owner_id
token_hash
status
is_current
assigned_at
activated_at
disabled_at
lost_at
replaced_at
replaces_card_id
replacement_card_id
created_at
updated_at
```

Raw tokens are deliberately absent from the persistence model. Diagnostic `repr` and `str` output also redact the token hash.

## Lifecycle

Persisted statuses are:

| Status | Owner | `is_current` | Public access | Terminal |
|---|---:|---:|---:|---:|
| `unassigned` | none | `false` | no | no |
| `assigned` | required | `true` | no | no |
| `active` | required | `true` | yes, in future lookup | no |
| `disabled` | required | `true` | no | no |
| `lost` | required | `false` | never | yes |
| `replaced` | required | `false` | never | yes |

Allowed transitions are:

- `unassigned -> assigned`
- `assigned -> active`
- `active -> disabled`
- `disabled -> active`
- `assigned|active|disabled -> lost`
- `assigned|active|disabled -> replaced`

`lost` and `replaced` cannot return to an operational state. Activation and disablement update one card only; they do not affect sibling cards or enforce a one-active-card-per-user rule. Lifecycle validation is centralized in `CardService`, with repository predicates providing atomic stale-write protection.

## Repository and service boundaries

`CardRepository` owns MongoDB persistence, typed reads, indexed queries, atomic single-document updates, duplicate-key translation, and replacement-reference writes. It provides:

- create unassigned card;
- find by ID, serial, or token hash;
- list all, current, or active cards for a user;
- assign a card;
- transition status, mark lost, or mark replaced;
- link bidirectional replacement references;
- execute an operation in a MongoDB transaction.

`CardService` owns secure identity generation, bounded collision retries, user-existence checks, allowed transitions, ownership checks, terminal-state enforcement, and replacement orchestration. Future routes and maintenance commands must depend on this service rather than mutating the repository directly.

Assignment requires an existing unassigned ownerless card and an existing user. It sets `owner_id`, `status=assigned`, `is_current=true`, and `assigned_at` without changing the serial, hash, or creation time.

## Replacement

Replacement creates a new serial and token, assigns the new card to the old card's owner with `status=assigned`, and does not activate it automatically. The new card stores `replaces_card_id`; the old card becomes `replaced`, non-current, and stores `replacement_card_id`.

The complete sequence runs inside a MongoDB transaction:

1. Create the new unassigned card.
2. Assign it to the existing owner.
3. Mark the old eligible card as replaced and non-current.
4. Link the old card to the replacement.
5. Commit all changes or roll back all changes.

Activation and disablement are single-document transitions and do not require a multi-document transaction.

## Indexes

The `cards` collection initializes these indexes:

- unique `serial`;
- unique `token_hash`;
- non-unique `owner_id`;
- non-unique `status`;
- non-unique `(owner_id, is_current)`;
- non-unique `(owner_id, status)`;
- non-unique `replaces_card_id`;
- non-unique `replacement_card_id`.

There is deliberately no unique index on `(owner_id, is_current=true)` or `(owner_id, status=active)`. Multiple current and active cards per user are valid.

Index initialization is idempotent and incompatible existing definitions fail visibly. The legacy `medical_profiles.public_access.token` index remains unchanged for compatibility; new card tokens do not use that field.

## Security and observability

Safe operational telemetry may include operation name, request/correlation ID, internal card ID, serial where justified, owner ID after assignment, status transition, result category, and duration.

Never log or persist:

- raw public tokens;
- token hashes;
- public URLs containing tokens;
- provisioning response bodies;
- medical data;
- cookies or authorization headers.

Card persistence does not expose card data through user, profile, JWT, or existing HTTP response models.

## Deferred consumers

The foundation is ready for, but does not implement:

- admin card provisioning and assignment HTTP APIs;
- user card listing, activation, and disablement APIs;
- anonymous token-hash public lookup;
- lost-card workflows and replacement APIs;
- QR rendering and NFC writing;
- legacy profile-token migration or removal;
- inventory, manufacturing references, shipping, scans, audit events, frontend controls, or deployment changes.

## Verification

From `emercard-backend/`:

```bash
uv run ruff check .
uv run ruff format --check .
uv run pyright
uv run pytest
EMERCARD_TEST_MONGODB_URI=<disposable-replica-set-uri> uv run pytest -m mongo
uv run python -m emercard.db.initialize
uv run python -m emercard.db.initialize
```

The MongoDB suite requires a disposable database. Replacement transaction coverage requires a replica-set-capable MongoDB. The two initialization runs must both succeed without dropping or rebuilding data.
